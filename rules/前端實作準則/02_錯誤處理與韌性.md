# 錯誤處理與韌性

> 用途：建立標準化的「錯誤攔截與呈現」機制，讓系統在網路不穩、API 報錯、SignalR 斷線時保持韌性，不讓使用者看見白畫面或卡死的儀表板。

技術棧前提：jQuery AJAX + 統一 `api.js` 封裝 + 後端 RFC 7807 ProblemDetails + SignalR（`withAutomaticReconnect`）+ 骨架屏。

---

## 1. 錯誤分類與處理層級

| 錯誤類型 | 來源 | 處理層 |
|---------|------|--------|
| API / HTTP 錯誤 | 後端回 4xx / 5xx | `api.js` 的 `ajaxError` 集中處理 + 呼叫端區域 catch |
| 業務邏輯錯誤 | 門檻越界、設備離線等 | 呼叫端用 Toast / 行內訊息 |
| 即時連線中斷 | SignalR / WebSocket 斷線 | 連線層自動重連 + 連線狀態提示 |
| 前端執行期例外 | JS 未捕捉錯誤 | `window.onerror` 集中捕捉 + console（見第 7 節） |

---

## 2. 統一 API 封裝（api.js）

所有對後端的請求必須走統一封裝，**不可在各模組裡隨手寫 `$.ajax`**。集中管理逾時、CSRF Token、錯誤處理。

```js
// api.js — 包一層 $.ajax，集中設定
const api = (function () {
  const DEFAULT_TIMEOUT = 8000; // 逾時第二道防線

  // 取 ASP.NET Core antiforgery token（由 Razor 產出的隱藏欄位，見 05）
  function antiforgeryToken() {
    return $('input[name="__RequestVerificationToken"]').val();
  }

  function request(options) {
    return $.ajax($.extend({
      timeout: DEFAULT_TIMEOUT,
      dataType: 'json',
      headers: { 'RequestVerificationToken': antiforgeryToken() }
    }, options));
  }

  return {
    get:  (url)        => request({ url, method: 'GET' }),
    post: (url, data)  => request({ url, method: 'POST', contentType: 'application/json', data: JSON.stringify(data) }),
    put:  (url, data)  => request({ url, method: 'PUT',  contentType: 'application/json', data: JSON.stringify(data) }),
    del:  (url)        => request({ url, method: 'DELETE' })
  };
})();
```

### 集中錯誤處理：ajaxError

用全域 `ajaxError` 接住所有沒被區域 catch 的 AJAX 失敗，統一解析 ProblemDetails 並呈現。

```js
// 集中錯誤處理：掛在 document 上，所有 $.ajax 失敗都會經過
$(document).ajaxError(function (event, jqXHR, settings, thrownError) {
  // 區域已自行處理者可在 settings 上標記 skipGlobalError 跳過
  if (settings.skipGlobalError) return;

  const problem = parseProblemDetails(jqXHR);
  const ui = mapStatusToUi(jqXHR.status, thrownError);

  if (ui.kind === 'toast') {
    Toast.show(ui.tone, problem.detail || ui.message);
  }
  // 行內錯誤由呼叫端自行處理（見對照表）
});
```

## 3. ProblemDetails 解析（RFC 7807）

後端統一回 RFC 7807 ProblemDetails，前端集中解析。

```jsonc
// 欄位驗證錯誤（400）
{ "type": "ValidationError", "title": "輸入資料驗證失敗", "status": 400,
  "errors": { "threshold": ["門檻需大於 0"] } }

// 系統錯誤（500）→ 只給 traceId，不暴露 stack trace
{ "type": "InternalError", "title": "系統發生錯誤", "status": 500, "traceId": "abc123" }
```

```js
function parseProblemDetails(jqXHR) {
  try {
    const p = jqXHR.responseJSON || JSON.parse(jqXHR.responseText);
    return {
      title: p.title,
      detail: p.detail || p.title,
      errors: p.errors || null,    // 欄位層級錯誤（驗證）
      traceId: p.traceId || null
    };
  } catch (e) {
    return { title: '未知錯誤', detail: '伺服器回應無法解析' };
  }
}
```

行內欄位錯誤呈現（取代框架的 `setError`）：把 `errors` 對應到欄位旁的訊息容器。

```js
function showFieldErrors(errors) {
  $('.field-error').text('');            // 先清空
  $.each(errors, function (field, msgs) {
    $('[data-error-for="' + field + '"]').text(msgs[0]); // 顯示第一筆
  });
}
```

## 4. HTTP 狀態碼 → UI 對照表（jQuery 版）

每種情境對應固定 UI，不可混用。判定邏輯集中在 `mapStatusToUi`。

| 狀態碼 | 情境 | 處理層 | UI 呈現 |
|--------|------|--------|---------|
| **400** | 欄位驗證失敗 | 呼叫端 | 行內錯誤（`data-error-for` 旁顯示），不跳轉 |
| **401** | 未授權 / Session 過期 | 集中 | Toast 紅「請重新登入」（內部系統可導回登入頁） |
| **403** | 權限不足 | 集中 | Toast 紅「您沒有此操作的權限」，留在原頁 |
| **404 (API)** | 資源不存在 | 呼叫端 | 行內提示「找不到資料」，不整頁崩潰 |
| **409** | 資料衝突 | 呼叫端 | 行內錯誤（靠近操作位置），顯示後端 `detail` |
| **422** | 語義錯誤（格式對但不合規） | 呼叫端 | 行內錯誤（同 400） |
| **429** | 請求過於頻繁 | 集中 | Toast 橘「請求過於頻繁，請稍後再試」 |
| **500** | 伺服器內部錯誤 | 集中 | Toast 紅「伺服器發生錯誤，請稍後再試」（不暴露細節） |
| **502** | 後端服務掛掉 | 集中 | Toast 紅「服務暫時無法連線」 |
| **503** | 系統維護中 | 集中 | Toast 橘「系統維護中，請稍後再試」 |
| **504** | 請求逾時 | 集中 | Toast 橘「請求逾時，請重新操作」 |

### 非 HTTP 錯誤

| 類型 | jQuery 判定 | UI 呈現 |
|------|------------|---------|
| 網路斷線 | `jqXHR.status === 0` | Toast 橘「網路連線異常，請確認網路狀態」 |
| 請求逾時 | `thrownError === 'timeout'` | Toast 橘「連線逾時，請稍後再試」 |
| 頁面資料載入失敗 | 呼叫端 catch | 行內 ErrorMessage + 重試按鈕 |
| 列表無資料 | 回傳空陣列 | EmptyState（說明文字，不可顯示空白區域） |

```js
function mapStatusToUi(status, thrownError) {
  if (status === 0)           return { kind: 'toast', tone: 'warning', message: '網路連線異常，請確認網路狀態' };
  if (thrownError === 'timeout') return { kind: 'toast', tone: 'warning', message: '連線逾時，請稍後再試' };
  switch (status) {
    case 400: case 404: case 409: case 422: return { kind: 'inline' };
    case 403: return { kind: 'toast', tone: 'danger',  message: '您沒有此操作的權限' };
    case 429: case 503: case 504: return { kind: 'toast', tone: 'warning', message: '請稍後再試' };
    default:  return { kind: 'toast', tone: 'danger', message: '伺服器發生錯誤，請稍後再試' };
  }
}
```

## 5. SignalR 斷線重連

即時推播是儀表板核心，斷線必須自動恢復並讓使用者看得到連線狀態。輪詢（jQuery AJAX）作為降級備案。

```js
// deviceFeed.js — SignalR 連線層，斷線自動重連 + 狀態提示
const DeviceFeed = (function () {
  let connection;
  let pollingTimer = null;

  function start() {
    connection = new signalR.HubConnectionBuilder()
      .withUrl('/hubs/vibration')
      .withAutomaticReconnect([0, 2000, 5000, 10000]) // 重試間隔
      .build();

    connection.on('deviceUpdated', function (device) {
      AppState.set('devices', mergeDevice(AppState.get('devices'), device));
      DeviceRenderer.patch(device); // 差異更新單張卡片
    });

    // 連線狀態提示
    connection.onreconnecting(() => ConnBanner.show('reconnecting', '連線中斷，重新連線中…'));
    connection.onreconnected(() => { ConnBanner.hide(); stopPolling(); });
    connection.onclose(() => { ConnBanner.show('offline', '即時連線已中斷'); startPolling(); });

    connection.start()
      .then(() => ConnBanner.hide())
      .catch(() => startPolling()); // 連不上 → 降級輪詢
  }

  // 降級備案：輪詢
  function startPolling() {
    if (pollingTimer) return;
    pollingTimer = setInterval(function () {
      api.get('/api/devices').done(list => list.forEach(DeviceRenderer.patch));
    }, 5000);
  }
  function stopPolling() { clearInterval(pollingTimer); pollingTimer = null; }

  return { start };
})();
```

要點：
- `withAutomaticReconnect()` 處理短暫斷線；徹底斷線（`onclose`）才切換到輪詢。
- 連線狀態用固定位置的 banner 呈現（重連中 / 已離線），重連成功立即關閉並停止輪詢。
- 重連成功後拉一次完整資料，補回斷線期間漏掉的推播。

## 6. 骨架屏與 EmptyState

- **載入中**：首次抓資料時先渲染骨架卡片（灰色佔位 + shimmer），避免空白與版面跳動（Layout Shift）。樣式見 03。
- **無資料**：API 回空陣列時顯示 EmptyState（圖示 + 說明文字），不可只留空白區域。
- **載入失敗**：區塊內顯示 ErrorMessage + 重試按鈕，只壞該區塊，不讓整頁崩潰。

```js
// 局部失敗隔離：圖表載入壞掉只影響圖表區
api.get('/api/devices/' + id + '/trend')
  .done(data => TrendChart.render(data))
  .fail(() => $('#trend-panel').html('<div class="error-box">趨勢載入失敗 <button type="button" class="btn-retry">重試</button></div>'));
```

## 7. 簡單前端錯誤捕捉

不需要 Sentry / LogRocket 這類重型 Telemetry（對內部面試專案過度設計）。只要一小段集中捕捉即可：

```js
// 全域未捕捉例外 → console 記錄（必要時加上送後端的精簡 endpoint）
window.onerror = function (message, source, line, col, error) {
  console.error('[未捕捉錯誤]', message, source + ':' + line + ':' + col, error);
  // 可選：api.post('/api/client-log', { message, source, line, col, ua: navigator.userAgent });
  return false; // 不阻止瀏覽器預設行為
};

// 未處理的 Promise rejection
window.addEventListener('unhandledrejection', function (e) {
  console.error('[未處理的 Promise]', e.reason);
});

// AJAX 錯誤已由第 2 節的 $(document).ajaxError 集中捕捉並 console
```

要點：以 `window.onerror` + `unhandledrejection` + jQuery `ajaxError` 三者集中捕捉，搭配 `console`。需要保留時可加一個精簡的 `/api/client-log` 端點，但不導入第三方服務。
