# 前端架構與可維護性

> 用途：定義本振動狀態監測儀表板的技術邊界、語意化 HTML/a11y、Razor Partial 與 JS 模組切分、以及簡單的前端狀態管理，讓專案從小長到大都好維護。

技術棧前提：ASP.NET Core 9 MVC（IIS / ASP.NET Core Module）+ Razor View / Partial View + 原生 JavaScript + jQuery + CSS（無 Tailwind）+ ECharts + SignalR。應用情境為內部用的振動監測儀表板（10~20 台設備，顯示振動值 mm/s、狀態、趨勢圖），非對外官網。

---

## 1. 技術邊界與 MVP 收斂

動工前先把範圍框住，避免「什麼都想做」：

- **MVP 必須有**：設備卡片列表（值、狀態燈、單位 mm/s）、即時更新（SignalR 推播）、點選設備看趨勢圖、狀態門檻對接 ISO 10816 分區（良好 / 異常 / 危險）。
- **Nice-to-have 延後**：複雜的歷史報表、匯出、誇張動畫、多主題切換。
- **技術妥協原則**：前端不做巨量資料運算。趨勢資料的降採樣、門檻判定、聚合一律推到後端 API 處理，前端只負責呈現。
- **不需要的東西（內部系統）**：無 SEO（不需被爬蟲索引）、無複雜登入流程、無 SSR/SSG。頁面由 MVC Razor 直接輸出即可。
- **瀏覽器支援**：以內部環境常見的現代瀏覽器為準（Edge / Chrome 近版），不需支援 IE11，可放心使用 CSS Grid、CSS 變數、`fetch`、ES6。

## 2. 語意化 HTML 與無障礙（a11y）

即使是內部系統，語意化標籤能讓畫面結構清楚、鍵盤可操作、維護更容易。

- **用對標籤，別全盤 `<div>`**：版面用 `<header>`、`<main>`、`<nav>`、`<section>`；設備卡片用 `<article>` 或 `<ul><li>`；標題依序 `<h1>` → `<h2>`，不可斷層。
- **`<button>` vs `<a>`**：
  - 「切換頁面 / 連結」用 `<a href>`。
  - 「開啟明細 / 觸發動作」用 `<button type="button">`。
  - 絕不用 `<div onclick>` 假裝按鈕（鍵盤、無障礙都會壞）。
- **狀態要可被讀出**：狀態燈不要只靠顏色傳達。除了顏色，補上文字（良好 / 異常 / 危險）或 `aria-label`，色盲使用者也能辨識。
- **即時數值用 `aria-live`**：數值會被 SignalR 動態更新的區域，可加 `aria-live="polite"`，讓螢幕報讀器知道內容有變。
- **圖片 / icon**：有意義的 `<img>` 補 `alt`；純裝飾的設 `alt=""`。

```html
<article class="device-card" aria-label="設備 P-12 振動狀態">
  <h2 class="device-card__name">P-12 主軸承</h2>
  <p class="device-card__value" aria-live="polite">4.2 <span class="unit">mm/s</span></p>
  <span class="status status--warning" role="status">異常</span>
  <button type="button" class="btn-detail" data-device-id="P-12">查看趨勢</button>
</article>
```

## 3. Razor Partial + JS 模組切分

對應「元件化」的等價作法：畫面用 Razor Partial 切，邏輯用 JS 模組物件封裝。

### 3.1 View 層：Partial View 切分

- 大頁面拆成可重用的 Partial：`_DeviceCard.cshtml`、`_StatusBadge.cshtml`、`_TrendChart.cshtml`、`_Skeleton.cshtml`。
- Partial 只負責「長相」，吃 ViewModel 渲染，不寫商業判斷邏輯（判斷推到後端或 JS 模組）。這等同前端的「展示型元件」概念。
- 容器頁（如 `Dashboard/Index.cshtml`）負責組合各 Partial、放置掛載點與初始資料，等同「容器型元件」。

```cshtml
@* Views/Dashboard/Index.cshtml — 容器：組合 + 提供掛載點 *@
<main id="dashboard">
  <section id="device-grid" class="device-grid">
    @foreach (var d in Model.Devices)
    {
        <partial name="_DeviceCard" model="d" />
    }
  </section>
</main>

@* 初始資料以 JSON 內嵌，避免首屏多打一次 API *@
<script id="initial-data" type="application/json">@Html.Raw(Model.DevicesJson)</script>
```

> 完整命名規範(JS 模組/變數、HTML id/data-\*、CSS BEM/變數、前後端橋接)見 [../命名規範.md](../命名規範.md)。

### 3.2 JS 層：IIFE / 模組物件封裝，職責分離

不要把變數和函式散落在全域。用模組物件封裝，內部狀態私有，只對外暴露必要方法。三個明確職責分開：

- **資料抓取模組**：負責呼叫 API / 接 SignalR。
- **渲染模組**：負責把資料畫到 DOM（差異更新，見 04 效能準則）。
- **圖表模組**：負責 ECharts 初始化與更新（延遲載入，見 04）。

```js
// dashboard.js — 容器模組，串接三個職責模組
const Dashboard = (function () {
  let initialized = false;

  function init() {
    if (initialized) return;
    initialized = true;

    // 1. 載入內嵌初始資料
    const seed = JSON.parse(document.getElementById('initial-data').textContent);
    AppState.set('devices', seed);

    // 2. 首次渲染
    DeviceRenderer.renderAll(seed);

    // 3. 啟動即時更新（SignalR，降級為輪詢見 02）
    DeviceFeed.start();
  }

  return { init };
})();

$(function () { Dashboard.init(); });
```

```js
// 渲染模組：只管畫面，不抓資料
const DeviceRenderer = (function () {
  function renderAll(devices) { /* 首次建立卡片 */ }
  function patch(device) { /* 差異更新單張卡片，見 04 */ }
  return { renderAll, patch };
})();
```

模組命名建議：`api`（API 封裝，見 02）、`AppState`（狀態，見下節）、`DeviceFeed`（SignalR/輪詢）、`DeviceRenderer`（渲染）、`TrendChart`（ECharts）。

## 4. 簡單的前端狀態管理（appState + 事件）

不需要任何狀態管理套件。用一個 **module 層級的狀態物件** + **自訂事件** 做跨模組通知，取代散落各處的全域變數。

### 運作機制

- **`set` 是唯一寫入閘門**：每次 `set` 把「改 `state`」和「廣播 `appState:change`」綁在一起，狀態一變訂閱者必定收到通知，不會出現「資料改了畫面卻沒更新」。
- **`document` 當廣播中繼站**：發布端 `trigger`、訂閱端 `on`，兩邊互不認識；要新增一個反應行為，只要再掛一個訂閱者，不必改寫發布端。
- 一次完整循環：點卡片 → `set('selectedId', id)` → 廣播 → 訂閱者收到後 `TrendChart.load(id)`。按鈕模組全程沒碰到 `TrendChart`，這就是解耦。

### 原則

- 所有共用資料集中在一個 `AppState` 物件，外部只能透過 `get` / `set` 存取。
- 狀態變更時，用 `$(document).trigger()` 廣播自訂事件，需要的模組用 `$(document).on()` 訂閱，達成模組解耦（A 模組不需要知道 B 模組存在）。
- 區分狀態種類：
  - **UI 狀態**（目前選中的設備、面板開合）→ 放 `AppState`。
  - **伺服器資料**（設備清單、趨勢資料）→ 也放 `AppState`，但**只由資料模組寫入、UI 模組只讀**（單一寫入者）。真相在後端，收斂寫入點才能保證畫面與後端一致、出錯時可追查是誰改的。
  - **單一函式內的暫存** → 用區域變數即可，不必進 `AppState`。
- 判斷準則：**會被別的模組讀取、或變更後需要通知別人**才進 `AppState`；只有自己函式用、跑完就丟的留區域變數。`AppState` 不是「所有狀態的垃圾桶」，亂塞會讓廣播變吵、狀態難追。

```js
// appState.js — module 層級的單一狀態來源
const AppState = (function () {
  const state = { devices: [], selectedId: null };

  function set(key, value) {
    state[key] = value;
    // 廣播：key 變更事件，跨模組通知
    $(document).trigger('appState:change', { key, value });
  }

  function get(key) { return state[key]; }

  return { get, set };
})();
```

```js
// 任何模組訂閱變更，彼此不直接耦合
$(document).on('appState:change', function (e, payload) {
  if (payload.key === 'selectedId') {
    TrendChart.load(payload.value); // 選中設備改變 → 載入該設備趨勢圖
  }
});
```

### 注意事項

- **單一事件 + key 過濾的代價**：目前所有變更共用 `appState:change`，訂閱者要自己 `if (key === ...)` 篩選。訂閱者一多時，可改用帶 key 的事件名（如 `appState:change:selectedId`），讓各模組只聽自己關心的。
- **`get` 回傳的是參考**：`AppState.get('devices').push(x)` 會偷改到內部狀態又不觸發廣播。要嚴謹就約定「拿到只讀」，或在 `set` 時存副本。

### 自訂事件命名

統一前綴避免衝突：`appState:change`、`device:updated`、`signalr:connected`、`signalr:reconnecting`。

## 5. 可維護性檢查清單

- [ ] 畫面以 Partial View 切分，Partial 不含商業判斷邏輯
- [ ] JS 以模組物件封裝，資料 / 渲染 / 圖表三職責分離
- [ ] 無散落的全域變數，共用狀態集中於 `AppState`
- [ ] 跨模組通知走自訂事件，不直接互相呼叫內部函式
- [ ] 語意化標籤正確，狀態不只靠顏色傳達
