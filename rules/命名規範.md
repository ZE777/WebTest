## 1. 後端(C#)

### 1.1 命名空間與資料夾

- 根命名空間 = 專案名 `VibrationDashboard`。
- 命名空間**與實體資料夾路徑一一對應**,多層用 `.` 串接:`Controllers/Api/` → `VibrationDashboard.Controllers.Api`。
- 一個資料夾代表一個職責分層;子目錄依「功能」再切(如 `Services/Machines`、`Services/Realtime`)。

| 資料夾 | 職責 | 內含命名模式 |
|---|---|---|
| `Controllers/` · `Controllers/Api/` | MVC 頁面 / REST API(薄層) | `{Name}Controller` |
| `Models/` | 核心領域模型(資料來源中立) | **無後綴**:`Machine`、`Measurement`、`MachineStatus`(enum) |
| `DTOs/Responses/` · `DTOs/Requests/` | 對外契約(邊界) | `{X}Dto` |
| `Data/Entities/` | EF Core 持久化實體 | `{X}Entity` |
| `Data/Configurations/` | EF 對映設定 | `{X}Configuration` |
| `Data/Migrations/` | EF 遷移(自動產生) | `{timestamp}_{Description}` |
| `DataSources/` · `/Json` · `/Sql` | 資料存取抽象 | `I{X}DataSource`、`{X}DataSource`、`{X}FileModel` |
| `Services/{Feature}/` | 業務邏輯 | `I{X}Service` + `{X}Service` |
| `Hubs/` | SignalR | `{X}Hub` |
| `Middleware/` | HTTP 管線 | `{X}Middleware` |
| `Common/Settings/` · `/Helpers/` · `/Exceptions/` | 橫切關注 | `{X}Options`、Helper(見 1.2)、`{X}Exception` |

### 1.2 類別 / 介面後綴(角色即後綴)

類別名 = `PascalCase` + **角色後綴**,看名字就知道它在哪一層:

| 後綴 | 角色 | 範例 |
|---|---|---|
| `Controller` | 控制器 | `MachinesController` |
| `Service` | 業務邏輯實作 | `MachineService` |
| `Dto` | 對外資料契約 | `MachineSummaryDto`、`MachineDetailDto`、`MeasurementDto` |
| `Entity` | EF 持久化實體 | `MachineEntity`、`MeasurementEntity` |
| `Configuration` | EF 對映設定 | `MachineConfiguration` |
| `Options` | 強型別設定 | `MachineDataOptions` |
| `Middleware` | 管線中介軟體 | `ExceptionMiddleware` |
| `Exception` | 自訂例外 | `AppException`、`NotFoundException` |
| `Hub` | SignalR Hub | `MachineHub` |
| `DataSource` | 資料存取實作 | `JsonFileDataSource`、`SqlMachineDataSource` |
| `FileModel` | JSON 檔案結構映射 | `MachineFileModel`、`MeasurementFileModel` |
| **無後綴** | 領域模型、回傳值物件、靜態工具 | `Machine`、`MachineImage`、`StatusEvaluator`、`MeasurementAnalyzer` |

- **介面一律 `I` 前綴**:`IMachineService`、`IMachineDataSource`、`IMachineNotifier`。
- **領域模型不加後綴**(`Machine`、`Measurement`)。
  > 註:本文件取代後端 01 §4「領域業務物件用 `XxxDomain` 後綴」的舊說法——實際程式碼**未使用** `Domain` 後綴,以本文件為準。
- **Helper 以功能命名,不強制 `Helper` 後綴**:靜態工具用語意化名稱 `StatusEvaluator`、`MeasurementAnalyzer`(而非 `StatusHelper`)。

### 1.3 方法命名

- **非同步**(回傳 `Task`/`Task<T>`)一律 `Async` 結尾;同步方法不加。
  - `GetSummariesAsync`、`GetDetailAsync`、`NotifyChangedAsync` ↔ `Evaluate`、`Analyze`(同步靜態工具)。
  - **例外:Controller action 不加 `Async`**(`GetAll`、`GetById`、`GetImage`)。action 名是路由/契約的一部分,且 ASP.NET Core 預設會移除 Async 後綴,故與框架慣例一致。Service/DataSource/Hub 等內部非同步方法仍需 `Async`。
- 動詞語意固定:

| 動詞 | 用途 | 範例 |
|---|---|---|
| `Get` | 取單筆 | `GetByIdAsync`、`GetImageAsync` |
| `GetAll` / `Get{X}s` | 取集合 | `GetSummariesAsync` |
| `Map` | 載體轉換 | `MapToMachine`、`MapToMachineEntity` |
| `Evaluate` / `Analyze` / `Compute` | 計算/判定 | `Evaluate`(狀態)、`ComputeSeverity` |
| `Notify` | 推播 | `NotifyChangedAsync`、`NotifyRemovedAsync` |
| `Invoke` | 中介軟體進入點 | `InvokeAsync` |

### 1.4 屬性 / 欄位 / 常數 / 區域變數

- **公開屬性** PascalCase,優先 `required` + `init`:`public required string Id { get; init; }`。
- **私有欄位** `_camelCase`:`private readonly IMachineService _service;`。
- **常數**以 PascalCase 為主:`public const string SectionName = "MachineData";`、`private const int DebounceMs = 400;`。
  - 唯讀集合用 `private static readonly`:`AllowedImageTypes`。
- **區域變數** camelCase、簡短達意:`var machines = ...;`、`var latest = m.Latest;`。

### 1.5 四種資料載體的分工(重要)

命名上嚴格區分四種資料載體,各自只活在自己的層,跨層必經 `MapTo*` 轉換:

```
JSON 檔  → MachineFileModel → Machine(領域) → MachineSummaryDto / MachineDetailDto → API/Hub
MS-SQL   → MachineEntity    ↗(MapToMachine)        ↗(ToSummary / ToDetail)
```

| 載體 | 後綴 | 活在哪層 | 不可做的事 |
|---|---|---|---|
| `*FileModel` | FileModel | `DataSources/Json` | 不外流到 Service 以上 |
| `*Entity` | Entity | `Data` / `DataSources/Sql` | **不可直接序列化給前端** |
| 領域模型 | 無 | `Models`(核心) | 不帶 ORM / JSON 屬性標註 |
| `*Dto` | Dto | `DTOs`(邊界) | 不含業務邏輯 |

> 已知刻意差異(由 `MapTo*` 配對,非疏漏):`Machine.ImageBytes` ↔ `MachineEntity.Image`;`Measurement.Time` ↔ `MeasurementEntity.MeasuredAt`(Entity 用更具體的欄位名對映資料庫)。

### 1.6 API 路由 / Controller / Action

- 路由屬性 `[Route("api/{resource}")]`,resource **全小寫、複數**:`api/machines`。
- 子資源用斜線:`[HttpGet("{id}/image")]` → `api/machines/{id}/image`。
- Action 依語意命名、**不加 `Action` 後綴**;HTTP 動詞用 attribute 明示:`[HttpGet]`、`[HttpGet("{id}")]`。
- Controller 標 `sealed`,繼承 `ControllerBase`(API)。

```csharp
[Route("api/machines")]
public sealed class MachinesController : ControllerBase
{
    [HttpGet]               public Task<ActionResult<IReadOnlyList<MachineSummaryDto>>> GetAll(...);
    [HttpGet("{id}")]       public Task<ActionResult<MachineDetailDto>> GetById(string id, ...);
    [HttpGet("{id}/image")] public Task<IActionResult> GetImage(string id, ...);
}
```

### 1.7 依賴注入 / Options

- 介面與實作配對 `I{X}Service` / `{X}Service`,介面與實作**同資料夾**。
- Options Pattern:`{Feature}Options` + 公開常數 `SectionName`,在 `Program.cs` 綁定設定區段。

```csharp
builder.Services.AddScoped<IMachineService, MachineService>();
builder.Services.Configure<MachineDataOptions>(
    builder.Configuration.GetSection(MachineDataOptions.SectionName)); // "MachineData"
```

### 1.8 EF Core / 資料庫

- Entity 屬性 PascalCase,對映欄位於 `{X}Configuration` 指定。
- 資料表名 = Entity 去 `Entity` 後綴(`MachineEntity` → 表 `Machine`)。
- 索引命名 `IX_{Table}_{語意}`:`IX_Measurement_Machine_Time`。
- `ErrorCode` 命名 `{RESOURCE}_{ACTION}`(全大寫底線):`MACHINE_NOT_FOUND`(見後端 02 §4)。

## 2. 前端(JS / CSS / Razor)

### 2.1 JS 檔案組織與模組

- 每個自寫檔對應**單一職責**,檔名 camelCase 以功能命名。
- 一律 **IIFE 封裝**;**功能模組掛單一全域物件**(PascalCase);共用工具檔(如 `api.js`)可掛多個工具(`Api`/`Toast`/`parseProblemDetails`)。**同為模組物件就同用 PascalCase**(含 `Api`),不因檔案而異。模組間透過全域物件與 `AppState` 自訂事件溝通,不互相直接呼叫內部函式。

| 檔案 | 職責 | 全域暴露 |
|---|---|---|
| `api.js` | AJAX 統一封裝 + 錯誤處理 + antiforgery | `window.Api`、`window.Toast` |
| `appState.js` | 單一狀態來源 + 自訂事件廣播 | `window.AppState` |
| `dashboard.js` | 容器:抓資料 → 寫狀態 → 渲染卡片/KPI | `window.Dashboard` |
| `trendChart.js` | 趨勢圖 Modal + ECharts 懶載入 | `window.TrendChart` |
| `realtime.js` | SignalR 推播 + 輪詢降級 | `window.Realtime` |
| `site.js` | 全站共用預留位 | —(目前空) |

### 2.2 JS 變數 / 函式 / 常數 / 事件

- **變數、函式** camelCase,函式採「動詞 + 名詞」:`buildCard`、`renderAll`、`applyUpdate`、`formatValue`。
- **模組層級常數** `UPPER_SNAKE_CASE`:`POLL_MS`、`DEFAULT_TIMEOUT`、`DAY_MS`;對照表常數可用 PascalCase 鍵(`STATUS_LABEL = { Good: '良好' }`)。
- **事件處理函式** `on` + PascalCase:`onMachineUpdated`;SignalR 內建回呼沿用其原名(`onreconnecting`、`onclose`)。
- **自訂事件名**統一前綴 `模組:動作`:`appState:change`、`device:updated`、`signalr:reconnecting`。

### 2.3 HTML `id` 與 `data-*`

- **`id` 用 kebab-case**:`kpi-total`、`trend-modal`、`alert-list`、`toast-root`。
- **`data-*` 屬性 kebab-case**(符合 HTML5):`data-id`、`data-range`、`data-close`。

### 2.4 CSS class(BEM)與 CSS 變數

- 自寫 class 一律 **BEM + kebab-case**:`Block`、`Block__Element`、`Block--Modifier`。
  - `card`、`card__media`、`card__name`、`card--selected`、`card--updated`。
- 狀態修飾符兩類:語意變體 `--good`/`--abnormal`/`--danger`;互動狀態 `is-active`。
- **CSS 變數 `--` + kebab-case**,集中在 `:root` 當 Design Token:`--bg`、`--surface`、`--accent`、`--space-3`、`--zone-danger`(細節見前端 03)。
- Bootstrap class 可與自寫 class 並存(`class="btn btn-ghost"`),但版面/卡片主樣式一律自寫 BEM,不混用第三方命名邏輯。

### 2.5 Razor / cshtml

- View 資料夾與檔名 **PascalCase**:`Views/Home/Index.cshtml`、`Views/Shared/`。
- **Partial / 共用樣板 `_` 前綴**:`_Layout.cshtml`、`_ViewImports.cshtml`、`_ValidationScriptsPartial.cshtml`;Partial 只負責長相,不寫業務判斷。
- 頁面標題統一 `ViewData["Title"]`;頁面專屬 JS 放 `@section Scripts { }`。

## 3. 前後端對接點(命名橋接)

| 介面 | 後端 | 前端接收 | 規則 |
|---|---|---|---|
| REST DTO 欄位 | `LatestValue`(PascalCase) | `latestValue`(camelCase) | 靠 `JsonNamingPolicy.CamelCase` **自動轉**,前端一律用 camelCase |
| SignalR 事件 | `MachineUpdated`(PascalCase) | `conn.on('MachineUpdated', …)` | **保持 PascalCase**——RPC 契約不走 JSON 序列化,前後端用完全相同字面 |
| API 路由 | `[Route("api/machines")]` | `api.get('/api/machines')` | 全小寫複數,前後端字面一致 |
| 錯誤回應 | ProblemDetails(`errorCode`/`traceId`) | 同一份解析 | JSON 欄位 camelCase;`errorCode` 值為 `MACHINE_NOT_FOUND` |

> 金句:「DTO 經序列化策略自動轉 camelCase,讓前端不必到處寫 PascalCase 變數;但 **SignalR 事件名刻意維持 PascalCase**,因為它是 RPC 方法名、不是 JSON 資料——這個差異是有意的契約設計,不是命名不一致。」

## 4. 已知的刻意例外(非疏漏,勿「修正」)

1. **載體欄位不同名**:`Machine.ImageBytes`↔`MachineEntity.Image`、`Measurement.Time`↔`MeasurementEntity.MeasuredAt`,由 `MapTo*` 配對(見 1.5)。
2. **Helper 無統一後綴**:`StatusEvaluator`、`MeasurementAnalyzer` 以功能命名而非 `*Helper`。
3. **常數風格雙軌**:後端 `SectionName`(PascalCase)、前端 `POLL_MS`(UPPER_SNAKE)——遵循各語言社群慣例,刻意不統一。
4. **id 用 kebab-case 但 JS 變數用 camelCase**:用途不同(DOM 標記 vs 程式識別字),無衝突。
5. **SignalR 事件 PascalCase**:見 §3,與前端 camelCase 變數慣例並存。
6. **Controller action 不帶 `Async`**:見 §1.3 例外。此為「controller action」自成一類且**類別內全部一致**(`GetAll`/`GetById`/`GetImage` 皆無),非同類別內不一致。

### 6.4 非命名範疇的旁註(供參考,不影響本次結論)

- `HomeController` 未標 `sealed`,其餘具體類別皆 `sealed`(`MachinesController` 等)。屬編碼慣例非命名規範;MVC Controller 慣例多不 sealed,可忽略或統一補上,擇一即可。

### 6.5 結論

命名規範與現行程式碼**一致**。同類別命名不一致(D2 `api`→`Api`)已修正;其餘(D1、D3)為不同類別的框架/結構慣例、各自類別內一致,已納入規則。未來實作以本文件為準。
