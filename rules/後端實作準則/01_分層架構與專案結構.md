# 後端分層架構與專案結構

> 本專案後端分層**參考既有專案的慣例**(Controller → Service → 資料層,搭配 Interfaces/DTOs/Common/Middleware),
> 並依本專案特性(MVC + Razor、JSON 檔為主要資料來源、SignalR 即時)做必要調整。
> 目的:對接團隊既有風格,並拿下「程式設計架構/可維護性」評分。

## 1. 資料夾結構(對映既有專案慣例並調整)

```
src/VibrationDashboard/
├─ Program.cs                  進入點:DI 註冊、Middleware 管線、SignalR/靜態檔
├─ appsettings.json            設定:DataSource:Type、監看資料夾路徑、門檻預設
├─ Controllers/                控制器(薄層,只負責 HTTP/路由)
│   ├─ HomeController.cs        Razor 看板頁
│   └─ Api/MachinesController.cs  REST API(清單/明細/圖檔)
├─ Hubs/                       SignalR
│   └─ MachineHub.cs
├─ Services/                   業務邏輯(介面與實作同資料夾,依功能分子目錄)
│   ├─ Machines/IMachineService.cs · MachineService.cs
│   └─ Realtime/FileWatcherService.cs(BackgroundService)
├─ DataSources/                ★資料存取抽象(本專案的「Repository 層」)
│   ├─ IMachineDataSource.cs
│   ├─ JsonFileDataSource.cs    主:讀資料夾 JSON
│   └─ SqlMachineDataSource.cs  可切換:MS-SQL(Phase 2)
├─ Models/                     核心領域模型 / EF 實體
│   ├─ Machine.cs · Measurement.cs · MachineStatus.cs(enum)
│   └─ Base/                    (SQL 路徑用)BaseEntity 等
├─ DTOs/                       API / Hub 對外契約(邊界資料)
│   ├─ Responses/              MachineSummaryDto · MachineDetailDto
│   └─ Requests/               (本專案讀取為主,需要才加)
├─ Common/                     橫切關注
│   ├─ Exceptions/             AppException + 子例外
│   ├─ Settings/               強型別設定類別(Options)
│   └─ Helpers/                共用工具(如門檻判定、測試資料產生)
├─ Middleware/                 ExceptionMiddleware → ProblemDetails
├─ Data/                       (SQL 路徑)AppDbContext + Configurations/
├─ SqlScripts/                 (SQL 路徑)Init / Seed
├─ Views/                      Razor 視圖 + Partial
├─ wwwroot/                    js / css / lib(echarts,jquery,signalr)
└─ App_Data/machines/          測試 JSON(10~20 份)
```

## 2. 各層職責與相依方向(單向,不可逆向相依)

```
Controllers / Hubs   ←只做 HTTP/推播,不含業務邏輯
      │  呼叫
      ▼
   Services          ←業務邏輯:狀態判定、組裝 DTO、協調流程
      │  透過介面
      ▼
 IMachineDataSource  ←資料存取抽象(JSON / SQL 可換)
      │
      ▼
  Models / Data      ←領域模型;SQL 路徑才用 EF/AppDbContext
```

**鐵則**
- Controller/Hub **薄**:接請求 → 呼叫 Service → 回 DTO。不直接碰資料來源或 DbContext。
- Service **不依賴具體資料來源**,只依賴 `IMachineDataSource` 介面 → 換 JSON/SQL 不影響 Service。
- **邊界一律用 DTO**,不把 EF 實體/內部模型直接序列化給前端(避免欄位外洩、降低耦合)。
- 相依只能由上往下,**禁止**下層反向引用上層。

## 3. 本專案相對既有專案的「刻意調整」(報告講點)

| 項目 | 既有專案 | 本專案 | 為什麼 |
|---|---|---|---|
| 資料存取 | Service 直接用 `AppDbContext`(EF) | Service 依賴 `IMachineDataSource` 介面 | 需求要「JSON 或 SQL」可切換,故多一層抽象 |
| App 類型 | 純 Web API | **MVC + Razor**(有 Views)+ API + SignalR | 題目要前端網頁 + 即時更新 |
| Domain 層 | `Domain/` 與 `Models/` 分離 | 規模小,`Models/` 即領域模型,邊界用 `DTOs/` | 避免過度設計;DTO-at-boundary 原則仍保留 |
| 即時 | 無 | 多 `Hubs/` + `FileWatcherService` | 需求 4 的即時更新 |

> 金句:「我沿用團隊的分層慣例,但因為這題要求資料來源可切換,我**在 Service 與資料之間多加了 `IMachineDataSource` 介面**——這是符合需求的抽象,不是為抽象而抽象。」

## 4. 介面與 DI 慣例(同既有專案)
- 介面與實作**同資料夾**:`Services/Machines/IMachineService.cs` + `MachineService.cs`。
- 介面命名 `IXxx`;領域模型**無後綴**(`Machine`、`Measurement`);對外 DTO 用 `XxxDto`。完整命名規範見 [../命名規範.md](../命名規範.md)。
- 在 `Program.cs` 用建構式注入註冊;資料來源依設定切換:
```csharp
builder.Services.AddScoped<IMachineService, MachineService>();
if (cfg["DataSource:Type"] == "Sql")
    builder.Services.AddSingleton<IMachineDataSource, SqlMachineDataSource>();
else
    builder.Services.AddSingleton<IMachineDataSource, JsonFileDataSource>();
builder.Services.AddHostedService<FileWatcherService>();
```

## 5. 寫入路徑與雙角色(管理者擴充)

唯讀(使用者)之外,管理者需要寫入(改門檻等)。寫入在分層上是**加法,不改動唯讀路徑**:

- **介面**:`IMachineDataSource` 增加寫入方法(或拆姊妹介面 `IMachineWriteStore` 做讀寫分離,ISP)。本專案規模小,先在同介面加 `UpdateThresholdsAsync(id, req)` 即可。
- **Request DTO**:寫入輸入放 `DTOs/Requests/`(如 `UpdateThresholdRequest`),與 Response DTO 對稱;Controller 用 DataAnnotations / ModelState 驗證。
- **Controller**:`Api/MachinesController` 加 `PUT /api/machines/{id}/thresholds`;管理頁由 `AdminController`(Razor)提供 `/admin`。**薄層原則不變**(驗證與商業規則在 Service)。
- **JSON 寫入安全**:`JsonFileDataSource` 採**原子覆寫**——先寫 `{id}.json.tmp` 再 `File.Move` 覆蓋,避免 watcher 讀到半截檔(見 [docs/03 §3](../../docs/03_資料來源_JSON與SQL設計.md))。
- **與即時路徑的綜效**:管理者存檔 → 檔案變更 → `FileWatcherService` 偵測 → SignalR 推播 → 所有看板自動更新。**寫入複用即時路徑,不另做通知**(設計創意講點)。
- **SQL 路徑(Phase 2)**:`SqlMachineDataSource` 寫入用 EF;schema 採 **Code-first + EF Migrations**(greenfield、結構簡單、自己掌控),`SqlScripts/` 只留 Seed,**不手寫建表 DDL**。

相依方向(補上寫入):
```
Controllers(/api PUT、/admin Razor)
   │ 呼叫
   ▼
Services  ←驗證(門檻 warning < danger)、組裝
   │ 透過介面
   ▼
IMachineDataSource(+ 寫入方法) → Json 原子覆寫 / SQL(EF, Code-first)
```

## 6. 檢查清單
- [ ] Controller/Hub 內沒有業務邏輯,只呼叫 Service。
- [ ] Service 只依賴 `IMachineDataSource`,沒有 `JsonFile`/`Sql` 具體型別。
- [ ] 對外回傳都是 `DTOs/`,沒有直接丟 EF 實體/內部模型。
- [ ] 相依方向單向;沒有下層引用上層。
- [ ] 切換 `DataSource:Type` 不需改 Controller/Service/前端。
- [ ] 寫入經 Service 驗證(門檻 warning < danger),Controller 不直接寫資料源。
- [ ] JSON 寫入採 tmp→rename 原子覆寫,不被 watcher 讀到半截檔。
- [ ] 管理端點(/admin)與唯讀端點(/)分離,唯讀路徑不受寫入影響。
