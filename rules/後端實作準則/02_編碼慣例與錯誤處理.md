# 後端編碼慣例與錯誤處理

> 沿用既有專案的編碼風格與「自訂例外 + 全域 Middleware → RFC 7807 ProblemDetails」機制。
> 這套錯誤格式與前端準則 `02_錯誤處理與韌性` 對接(前端解析同一份 ProblemDetails)。

## 1. 專案設定(.csproj)
```xml
<TargetFramework>net9.0</TargetFramework>   <!-- 本機為 .NET 9 -->
<Nullable>enable</Nullable>
<ImplicitUsings>enable</ImplicitUsings>
<GenerateDocumentationFile>true</GenerateDocumentationFile>  <!-- 供 Swagger 讀 summary -->
<NoWarn>$(NoWarn);1591</NoWarn>
```
- 套件:`Microsoft.EntityFrameworkCore.SqlServer`(SQL 路徑)、`Serilog.AspNetCore`、`Swashbuckle.AspNetCore`、`Microsoft.AspNetCore.SignalR`(內建)。

## 2. 命名與註解
> 完整命名規範(類別後綴、載體分層、前後端橋接)見 [../命名規範.md](../命名規範.md);本節僅列要點。
- 類別/方法/屬性 **PascalCase**;私有欄位 `_camelCase`;介面 `IXxx`。
- 公開型別與方法寫 **XML 文件註解**(`/// <summary>`),供 Swagger 與維護者閱讀(可用繁中)。
- EF 實體屬性可加 `<remarks>` 標註欄位型別/約束(對映資料庫),例如 `nvarchar(100) · NOT NULL`。
- 非同步方法以 `Async` 結尾並回傳 `Task`。

## 3. 資料存取慣例(SQL 路徑,沿用既有專案慣例)
- **讀取一律 `AsNoTracking()`** + `Select` 投影到 DTO/領域物件(不追蹤、不過抓欄位)。
- 主鍵用 **GUID**,由應用層產生(`Guid.NewGuid()`),不依賴資料庫自增。
- 時間統一在 `AppDbContext.SaveChangesAsync` 設定 `DateTime.Now`(台灣 CST+8),不散落各處。
- 「取某台最近 N 筆」靠索引 `(MachineId, MeasuredAt DESC)`(見 docs/03)。

> JSON 路徑(`JsonFileDataSource`):讀檔用 `System.Text.Json` 反序列化;寫測試檔先寫 `*.tmp` 再 `rename`(原子覆蓋),避免讀到半截檔。

## 4. 例外處理機制(核心)

**自訂例外**:所有業務例外繼承 `AppException`(帶 `StatusCode`、`ErrorCode`、`Details`)。不繼承它的例外視為非預期 → 500。

```csharp
// Common/Exceptions/AppException.cs
public class AppException : Exception
{
    public int StatusCode { get; }
    public string ErrorCode { get; }   // 命名:{RESOURCE}_{ACTION},如 "MACHINE_NOT_FOUND"
    public object? Details { get; }
    public AppException(string message, int statusCode, string errorCode, object? details = null)
        : base(message) { StatusCode = statusCode; ErrorCode = errorCode; Details = details; }
}

// 子例外:NotFoundException(404) / ValidationException(400) / ForbiddenException(403) ...
public class NotFoundException : AppException
{
    public NotFoundException(string resource, object key)
        : base($"{resource} 不存在 (key={key})", 404, $"{resource.ToUpper()}_NOT_FOUND") { }
}
```

**全域 Middleware**:`ExceptionMiddleware` 排在管線**最前面**,攔截所有例外,統一輸出 ProblemDetails:
- `AppException` → 用其 `StatusCode` / `Message` / `ErrorCode`。
- EF `DbUpdateException`(UNIQUE/FK)→ 409 / 400 對應錯誤碼。
- 其他未預期 → 500(只給 `traceId`,**不洩漏 stack trace**)。
- 擴充欄位:`errorCode`、`traceId`、(驗證錯誤時)`errors`;JSON 用 **camelCase**。

```jsonc
// 統一錯誤回應(RFC 7807)
{
  "status": 404,
  "title": "Machine 不存在 (key=PUMP-A01)",
  "detail": "...",
  "instance": "/api/machines/PUMP-A01",
  "errorCode": "MACHINE_NOT_FOUND",
  "traceId": "00-abc...-01"
}
```

**Service 層用法**:直接 `throw new NotFoundException("Machine", id);`,不在 Service 寫 try/catch 包 HTTP 細節——交給 Middleware。

## 5. 記錄(Logging)
- 用 **Serilog**(Console + File)。業務例外記 `Warning`、未預期記 `Error`(含 traceId 與請求路徑)。
- 關鍵流程記資訊級 Log(如:偵測到檔案變更、推播某台、狀態轉危險)。
- **不要**把機敏資料寫進 Log。

## 6. 檢查清單
- [ ] `ExceptionMiddleware` 已註冊且在管線最前面。
- [ ] 業務錯誤都用 `AppException` 子類,不在 Controller 散寫 try/catch。
- [ ] 對外錯誤是 ProblemDetails(camelCase,含 errorCode/traceId),500 不洩漏細節。
- [ ] 讀取查詢用 `AsNoTracking` + 投影。
- [ ] 公開型別有 XML 註解,Swagger 可讀。
