# 02 系統架構設計

> 對應評分標準 1:**程式設計架構、可維護性**。這份是報告時最該講清楚的一份。

## 1. 技術棧與選型理由

| 層 | 技術 | 為什麼 |
|---|---|---|
| 執行平台 | **IIS 10**(ASP.NET Core Module 反向代理到 Kestrel) | 題目指定 IIS;職缺要 IIS 上線能力 |
| 後端 | **ASP.NET Core 9 MVC + Web API** | 職缺要 ASP.NET Core/C#;現代、跨平台、好測試 |
| 即時 | **SignalR** | 檔案變更時主動推播,低延遲、只推 delta |
| 資料 | **JSON 檔(主)/ MS-SQL(可切換)** | 需求 4 是檔案監看情境;SQL 展示資料庫能力 |
| 前端 | **Razor + jQuery + 原生 JS** | 對接職缺技能(HTML5/CSS/JS/jQuery) |
| 圖表 | **Apache ECharts** | 內建可移動游標(axisPointer)、門檻線、分區色帶,最貼近附圖 |

> **為什麼不用 .NET Framework 的傳統 ASP.NET MVC?** 本機只裝了 .NET 9 SDK;ASP.NET Core 更現代且同樣能掛 IIS。介面、DI、BackgroundService、SignalR 都是內建一等公民。

## 3. 核心設計:資料來源抽象(可維護性的關鍵賣點)

用一個介面把「資料從哪來」與「業務/呈現」解耦,符合**策略模式 + 依賴反轉**:

```csharp
public interface IMachineDataSource
{
    Task<IReadOnlyList<Machine>> GetAllAsync(CancellationToken ct = default);
    Task<Machine?> GetByIdAsync(string id, CancellationToken ct = default);
}
```

用設定檔 + DI 決定要注入哪一個實作:

```jsonc
// appsettings.json
"DataSource": { "Type": "Json" }   // "Json" 或 "Sql"
```
```csharp
// Program.cs(節錄)
if (cfg["DataSource:Type"] == "Sql")
{
    builder.Services.AddDbContext<AppDbContext>(o => o.UseSqlServer(cfg.GetConnectionString("Default")));
    builder.Services.AddScoped<IMachineDataSource, SqlMachineDataSource>();   // 依賴 DbContext → Scoped
}
else
    builder.Services.AddSingleton<IMachineDataSource, JsonFileDataSource>();  // 無狀態讀檔 → Singleton
```

> **生命週期**:JSON 無狀態讀檔 → Singleton;SQL 依賴 `AppDbContext`(不可單例)→ Scoped。唯一消費者 `MachineService` 為 Scoped,兩者皆相容。
> **切換設定放哪(appsettings vs web.config)**:見 [08 §H-Q10](08_面試報告.md)——以 appsettings 為主、web.config 僅作 IIS 運維覆寫。

> **報告金句**:「換資料來源只要改一行設定,不用動到 Controller、Hub 或前端——這就是可維護性。」

## 4. 即時更新管線(細節見 [04](04_即時更新_輪詢與SignalR.md))
1. `FileWatcherService`(BackgroundService)監看 JSON 資料夾的 `Created/Changed/Deleted`。
2. 事件**去抖**(避免寫入中途觸發)後,重讀「那一台」。
3. 透過 `MachineHub` 推播該台 delta 給所有連線的瀏覽器。
4. 前端收到後**只更新那張卡片**(差異更新,不全量重建 DOM)。

## 5. API 端點(初步)

| 方法 | 路徑 | 用途 | 備註 |
|---|---|---|---|
| GET | `/api/machines` | 初次載入全部設備摘要 | 不含 base64 大圖 |
| GET | `/api/machines/{id}` | 單台明細(含最近 40 筆量測) | 點擊時懶載入 |
| GET | `/api/machines/{id}/image` | 機器圖檔(二位元) | **ETag/304 快取**,平時不重傳 |
| Hub | `/hubs/machine` | SignalR 推播 | `MachineUpdated` / `MachineRemoved` |
| GET | `/api/machines?search=&status=&sort=&page=&pageSize=` | 清單查詢 / 分頁 | **規劃**:現走前端篩選;規模大時啟用(見 [07 §5.3](07_領域資料與BI分析規劃.md)) |
| GET | `/api/machines/stats` | 全廠彙總(KPI / 分布 / 風險排行) | **規劃**:分頁後彙總改吃此端點,不再前端全量 rollup |

> 後兩列是**為規模化預留的接縫**:本專案 10~20 台不實作,但**契約先定**,上層用 `listQuery` 與彙總來源抽象隔離——未來切後端分頁是抽換實作、不動 UI(分頁與即時的互動見 [04 §8](04_即時更新_輪詢與SignalR.md))。

## 6. 專案結構(規劃)

分層**參考既有專案**的慣例(Controller → Service → 資料層,搭配 DTOs/Common/Middleware),
並依本專案調整(詳細規範見 [rules/後端實作準則/01](../rules/後端實作準則/01_分層架構與專案結構.md)):

```
src/VibrationDashboard/
  Program.cs · appsettings.json
  Controllers/   HomeController(Razor)、Api/MachinesController
  Hubs/          MachineHub(SignalR)
  Services/      Machines/IMachineService+MachineService、Realtime/FileWatcherService
  DataSources/   IMachineDataSource、JsonFileDataSource、SqlMachineDataSource  ← 資料存取抽象
  Models/        Machine、Measurement、MachineStatus(enum)、Base/(SQL 用)
  DTOs/          Responses/(MachineSummaryDto、MachineDetailDto)
  Common/        Exceptions/(AppException…)、Settings/、Helpers/
  Middleware/    ExceptionMiddleware → ProblemDetails
  Data/          AppDbContext + Configurations/(SQL 路徑)
  SqlScripts/    Init / Seed(SQL 路徑)
  Views/         Razor 視圖 + Partial
  wwwroot/       js / css / lib(echarts,jquery,signalr)
  App_Data/machines/   ← 測試 JSON(10~20 份)
```

**關鍵調整**:既有專案的 Service 直接用 `AppDbContext`;本專案因需求要「JSON 或 SQL 可切換」,
在 Service 與資料之間多一層 `IMachineDataSource` 介面(符合需求的抽象,非為抽象而抽象)。

## 7. IIS 部署重點
- 安裝 **ASP.NET Core Hosting Bundle**(提供 ASP.NET Core Module v2)。
- `dotnet publish` → IIS 建站台指向發行資料夾;Application Pool 設為「**No Managed Code**」(因為 Kestrel 跑 .NET,IIS 只當反向代理)。
- `web.config` 由發行自動產生;另加**安全標頭/CSP**(見前端資安準則)。
- SignalR 需 IIS 啟用 **WebSocket** 功能。

## 8. 開發期建置:Razor 執行階段編譯(**開發限定**)

預設 Razor View 是**編譯期**編進組件的(效能最佳)。開發時為了「改 `.cshtml` 不必重建即時生效」,加入 `Microsoft.AspNetCore.Mvc.Razor.RuntimeCompilation`,並**只在開發環境**啟用:

```csharp
// Program.cs(節錄)
var mvc = builder.Services.AddControllersWithViews().AddJsonOptions(/* … */);
if (builder.Environment.IsDevelopment())
    mvc.AddRazorRuntimeCompilation();   // 僅 Development;Production 維持編譯期 View
```

- **上機零影響**:`IsDevelopment()` 取決於 `ASPNETCORE_ENVIRONMENT`;Production 不會呼叫,View 走預編譯版本、無 Roslyn 即時編譯負擔。套件 DLL 雖隨輸出部署,但不被啟用(無害);潔癖可在 csproj 用 `Condition="'$(Configuration)'=='Debug'"` 只在 Debug 引用。
- **解決的坑**:用 `dotnet run --no-build`(或未重建)時,執行中的 app 仍跑**上次編譯的 Razor View**;若 `.cshtml` 改過(例如新增 `<script>` 引用一支前端依賴),會載不到新依賴而讓前端**靜默中斷**。執行階段編譯讓 `.cshtml` 變更即時生效,避免此類問題。**注意 `.cs` 變更仍需重建重啟。**

> **報告金句**:「開發期用 Razor 執行階段編譯換『改 View 即時生效』的回饋速度,但用 `IsDevelopment()` 把它**鎖在開發環境**——上機效能與行為跟沒加之前一模一樣。」
