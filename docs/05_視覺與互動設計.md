# 05 視覺與互動設計

> 對應評分標準 2(視覺美化)與 3(設計創意)。
> 方向:**FineReport 風 + 僅 4 色調 + 強調需觀覽資料 + 強化動態連動**。

## 1. 設計原則:色彩當「訊號」,不是裝飾
「僅 4 色調」與「狀態有 3 種(良好/異常/危險)」會打架,因為狀態天然要綠/黃/紅。
解法:**畫面絕大部分只有深色底 + 中性灰 + 一個主色,把綠黃紅留給「狀態與越線處」**。
因為四周幾乎無色,這些語意色會非常跳——同時滿足「克制用色」與「3 態語意」。

## 2. 色票(Design Tokens,集中用 CSS 變數)
```css
:root{
  --bg:        #0E1726;          /* 頁面底:深藍工業 */
  --surface:   #16243B;          /* 卡片/面板 */
  --surface-2: #1E3050;          /* 內層/hover */
  --text:      #C7D2E0;          /* 主文字 */
  --text-dim:  #7C8BA1;          /* 次要文字 */
  --grid:      rgba(255,255,255,.06); /* 格線/分隔 */
  --accent:    #22D3EE;          /* 主色:游標/選中/趨勢線(平時唯一彩色) */
  --good:      #22C55E;          /* 良好(綠) */
  --warning:   #F59E0B;          /* 異常(琥珀) */
  --danger:    #EF4444;          /* 危險(紅) */
}
```
讓使用者「感受到」的克制色票 = **深藍 / 青 / 灰**,語意色只在該強調時出現。
**禁止**在程式碼到處 hardcode 色碼,一律引用變數。

## 5. 趨勢圖規格(ECharts,對應需求 6 與附圖)
- **可移動游標**:`axisPointer`(type: `line` / `cross`),滑鼠移動即顯示該時間點數值。
- **tooltip**:顯示精確「時間 + 數值 + 單位 + 狀態」。
- **門檻線**:`markLine` 畫警告(黃)、危險(紅)兩條水平線。
- **分區色帶**:`markArea` 在背景畫良好/警告/危險三段色帶(極淡),強化「現在落在哪一區」。
- **越線資料點強調**:超過門檻的點變色放大(綠點→黃/紅點),呼應「強調需觀覽資料」。
- **X 軸(類目軸)**:量測是離散的**每日讀數**,X 軸用 `type:'category'`(每筆一格、等距)而非 `time`——讓資料點**精準對齊**下方日期標籤(`time` 軸會把刻度切在午夜整點、與 08:00 的點錯位),且刻度數量**隨選取資料變動**;多點時 `axisLabel.hideOverlap` 自動疏化標籤。
- **時間區間**:四段切換 **全部 / 7 天 / 30 天 / 自訂**,**預設「30 天」**;「自訂」展開起迄日期框(`<input type="date">`,`min`/`max` 夾在該設備資料跨度內、改值即重畫)。對應附圖右上的「7 days / 全部」切換。
- **Y 軸縮放(值差大時看細節)**:`dataZoom` 掛在 Y 軸——右側細滑桿拉出想看的數值帶、圖內滾輪縮放 / 拖曳平移、雙擊還原全覽。關鍵設 `filterMode: 'none'`:只改可視範圍、**不過濾**視窗外的點(否則趨勢線會斷),門檻線與分區色帶照常顯示。
- **區間統計 + 計算解釋 tooltip(怎麼算的)**:讀數頭的**趨勢 / 平均 / 最大 / 標準差 / 超標**會**隨上方選取區間重算**(切「7天」就只統計這 7 天;前端 `computeStats` 鏡像後端 `MeasurementAnalyzer`,見 [07 §5.2](07_領域資料與BI分析規劃.md))——圖與統計**看同一段資料**;**狀態 / 最新值**例外,固定取最新一筆(設備當前狀態)。每個值掛 `data-tooltip` 說明公式(趨勢=最小平方法回歸斜率、標準差=母體公式 √(Σ(值−平均)²÷N)、超標=達警告門檻**含 ≥** 的筆數),`cursor:help` 暗示;自訂區間無資料時顯示「此區間 無量測」。健康率走勢的 info icon tooltip 同此精神(見 §7.2)。

> **新增決議(2026-06-24,Phase 6 微調)— Y 軸縮放**:動機是微振監控常見「平常 0.3 mm/s、尖峰衝到 10」的資料,線性軸會把低值區壓成貼底一條線,看不出每日變化、也看不出離門檻多遠。
> **取捨:選 `dataZoom` 而非對數軸**——本看板以「值**相對門檻**(警告/危險)」為核心判讀,log 軸會扭曲門檻相對高低、且不能含 0;`dataZoom` 保留現有門檻線與分區色帶的視覺語言,只縮放可視範圍。
> 全用 ECharts **原生**元件(bundle 已含 `dataZoom`,零額外依賴);滑桿採**細軌 + accent 圓點把手**融入暗色主題。實作見 [trendChart.js](../src/VibrationDashboard/wwwroot/js/trendChart.js) 的 `buildOption`(`dataZoom` 設定)與 `renderChart`(雙擊還原)。

> **新增決議(2026-06-24,Phase 6 微調)— 時間區間預設與廣度**:把原本「7 天 / 全部」兩段擴成 **全部 / 7 天 / 30 天 / 自訂**,**預設落在「30 天」**。
> **理由(雙目的)**:這張趨勢圖同時承擔兩種看法 —— **近期即時觀察**(它最近怎樣、這幾天動向)用「週/月」、**長期趨勢與事件回溯**(劣化曲線、查特定時段)用「全部/自訂」。監管系統的主要決策是「**現在要不要處理**」,故預設權重放近期(30 天);但保留「全部 / 自訂」讓需要回溯歷史的人也能篩出特定片段——**不犧牲能力、也不讓介面變繁雜**。
> **取捨:不放「年/季」**——目前資料粒度是**每日一筆**,「全部」已涵蓋長期視圖、年/季屬冗餘;且資料尚未跨年,放了點下去與「全部」幾乎相同甚至空白,反而像壞掉。原則:**預設區間要對齊資料的實際跨度與粒度**。
> **實作與演進**:過濾仍在前端(日資料量級整段撈回即可;一年也才約 365 點,ECharts 吃得下);API **預留**可加 `from/to` 區間參數 + 後端降採樣,待粒度變高(時/分採樣)再切。程式見 `filterByRange`/`effectiveCustom` 與 [_TrendModal.cshtml](../src/VibrationDashboard/Views/Home/_TrendModal.cshtml) 的 `.seg__btn` + `#trend-custom`。
> **「近 N 天」用筆數算(避免 fencepost)**:相對區間直接取**最近 N 筆**(從最新往回數 N 筆、**含最新**),每日一筆 → 即第 N 天(`filterByRange` 的 `ms.slice(-N)`)。**為何用筆數而非時間視窗**:時間視窗 `[最新−N天, 最新]` 兩端含、邊界又剛好壓在 08:00 取樣點 → 7 天會含 **8** 筆(7 段間隔 = 8 個樁)、30 天含 31 筆;改取「最近 N 筆」保證**剛好 N 筆**、最直覺。此法也與**健康率走勢回推**一致——都「依序列位置從最新端對齊、不讀日期」(見 [07 §6.1](07_領域資料與BI分析規劃.md))。`自訂` 例外:使用者明確點選起迄 → 用**日期閉區間**(兩端含,端點補當日 00:00–23:59 並夾進資料跨度)。

> **新增決議(2026-06-24,Phase 6 微調)— X 軸改用類目軸(`time` → `category`)**:動機是兩個軸問題,根因同一個——量測是**每日離散讀數**,但 `time` 軸把刻度切在「nice」時間點(午夜整點)、且小範圍時硬切固定刻度數。
> **症狀**:① 資料點(08:00)落不到日期標籤(00:00)上,看起來沒對齊;② 自訂只選 6/16–6/17 時,軸仍切 7 格 → 標籤重複成「6/16 6/16 … 6/17」、不隨選取變動。
> **取捨:`type:'category'`(每筆一格、`boundaryGap:false`)**——資料規律(每日),等距類目與真實時間視覺等價,卻換來「點對齊標籤」+「刻度數=選取的資料筆數」;40 天等多點時 `axisLabel.hideOverlap` 自動疏化,不再硬切固定刻度。代價:若未來資料變**不等距**(時/分混採),類目軸會失真,屆時改回 `time` 軸 + 自訂刻度。tooltip 改取 `axisValue`(類目=timestamp)還原完整時間。實作見 [trendChart.js](../src/VibrationDashboard/wwwroot/js/trendChart.js) `buildOption` 的 `xAxis`。

### 7.1 告警事件清單 — 事件認定與互動規則

定位:這是一條**「設備狀態轉移」的事件流**(剛剛發生了什麼變化),**不是**「目前有幾台異常」的計數(那是 KPI 的事)。資料來源 = SignalR 推來的單台 delta,在前端累積(`Dashboard.Alerts`,見 [dashboard.js](../src/VibrationDashboard/wwwroot/js/dashboard.js))。

**事件認定規則(什麼才記一筆)**

| 情況 | 是否記錄 | 為什麼 |
|---|---|---|
| 狀態**升級**(正常→警告→危險) | ✅ 記一筆,標「升為 XX」 | 惡化要被看見 |
| 狀態**恢復**(危險→警告→正常) | ✅ 記一筆,標「恢復為 XX」 | 復原也是事件 |
| **首次出現**(`prevStatus == null`) | ❌ 不記 | 初次載入若每台都灌一筆會洗版 |
| 狀態**無變化**(`prevStatus === status`) | ❌ 不記 | 同態反覆推播不該洗版 |

→ 推前狀態 `prevStatus` **取自 `AppState` 全量(非當前頁 DOM)**:分頁後該台可能不在當前頁、無 DOM 節點,但升級/恢復是**全廠事件**,不能漏記(與 [07 §5.4](07_領域資料與BI分析規劃.md) 同一條鐵則)。

**後端不知道「告警」這回事(純前端衍生、不落地)**:SignalR 通道送的只是「這台**現在**長這樣」的單台摘要(狀態＋最新值),**沒有升級/恢復欄位**;對照組(前一次狀態)不在通道裡,而是存在前端 `AppState` 記憶體。所以「告警」是前端比對前後狀態算出來的衍生產物——**換成輪詢 API 也能算出同樣結果**,因為判定邏輯不在傳輸層。也因此告警**不落地**:SQL 模式下 DB 只有 `Machines`、`Measurements` 兩張表,沒有告警表;事件只活在瀏覽器(上限 50 筆),重整或清除就沒了。正式系統才會把告警寫進專門的告警表,並帶確認/已讀/靜音/升級通知等狀態機——這是 demo 範圍的刻意取捨。

**清單行為**
- **筆數徽章**:標題旁顯示目前累積筆數;空清單顯示「尚無告警事件」。
- **上限 50 筆、最新在上**(`unshift` + 截斷),避免無限增長。
- **清除**:一鍵清空——僅清前端累積的事件列,**不影響設備真實狀態**。
- **點某一筆 → 選取該設備**(高亮卡片 + 開趨勢 Modal),呼應 §6 動態連動。
- **載入態一致**:初次載入給 skeleton 行、載入失敗給精簡失敗行,與其他區塊相同的回饋節奏。
- **防 XSS**:設備名等外部字串一律走 `.text()`/`.attr()`,絕不進 `innerHTML`。

> 一句話:**告警清單記的是「變化」不是「現況」——首次與無變化不記、全廠不漏、最新 50 筆,點一筆直接跳到那台趨勢圖。**

### 7.2 狀態分布面板 — 可點擊篩選 + 健康率走勢

這塊(`.panel--dist`)除了三色占比長條,另含兩個互動 / BI 強化:

**① 長條段可點擊篩選**(KPI 那排之外的便利入口)
- 點長條的良好 / 異常 / 危險段 → 篩選下方清單只看該狀態(再點同段取消),**複用 KPI 與 `#filter-status` 的同一條接縫**(`trigger` 對應 seg 鈕),不另開邏輯。
- 選中回饋**不加框**(長條 `overflow:hidden` 會裁切邊框):改成**選中段保持滿亮、其餘兩段變淡**(`opacity .35`)。
- a11y:`role=button` + 依占比管理 `tabindex`(占比 0 的段退出 tab 序、對讀報器隱藏);鍵盤 `:focus-visible` 給細指示。

**② 健康率走勢 sparkline + 變化箭頭**
- 面板底部(`margin-top:auto` 沉底,補滿與風險排行等高後的留白)放一條**固定 0–100 縱軸**的迷你折線:線越高 = 越健康。
- 一載入就有歷史線(**用既有量測回推**,演算法見 [07 §6.1](07_領域資料與BI分析規劃.md)),之後每次更新追加一點。
- 右上箭頭比**最後一期 vs 前一期**:改善綠↑、惡化紅↓、持平灰→(語意色只留給「方向」,折線本身用主色 accent)。
- **計算解釋 tooltip**:走勢標題後放一顆 **info icon(`.info-dot`)**,滑到 / focus 到 icon 才跳 `data-tooltip`(不整列觸發),內容對齊 [07 §6.1](07_領域資料與BI分析規劃.md) 公式:`健康率 % = round(良好台數 ÷ 總台數 × 100)`、「良好」逐台用各自門檻判定(非加總健康度)、走勢初次回推+之後彙總追加、全廠口徑不受篩選/分頁影響——把口徑點破,避免被誤認為「當前頁占比」。

> 一句話:**長條多一個「點了就篩」的便利入口;面板底部把「現在多健康」升級成「正在變好還是變壞」。**

## 8. 微互動紀律(別吃效能)
- 能用 CSS `transition`/`animation` 就**不要用 JS**;動畫只改 `transform`/`opacity`(GPU 加速,不觸發重排)。
- 載入用**骨架屏**而非空白,避免版面跳動。
- 游標移動 / 視窗 resize 用 throttle/debounce。
