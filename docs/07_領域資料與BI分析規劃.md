# 07 領域資料設計與 BI 分析規劃

> 視角:**資料規劃/分析師**。回答三件事——這個振動監測領域要餵**什麼樣的假資料**、看板該**呈現哪些項目**、以及怎麼**篩選與分析**(含輕量 BI)。
> 上承 [01 需求分析](01_需求分析與領域背景.md)(資料模型、雙角色)與 [03 資料來源](03_資料來源_JSON與SQL設計.md)(JSON/SQL、正規化)。

## 1. 這份文件要解決的問題

題目只給了「機器名稱/圖檔/狀態/10 筆數值」的骨架。但一個**看起來像真的監控軟體**,差別不在欄位多寡,而在:

1. **假資料有沒有「故事」**——能不能讓趨勢圖、風險排行、KPI 一打開就有看頭。
2. **呈現的項目是否 actionable**——使用者一眼知道「該先看哪台、為什麼」,而不是看一堆數字。
3. **分析維度是否成立**——能不能合理地篩選、排序、彙總。

---

## 2. 關鍵前提:跨機**不能**比原始 mm/s

這是整份 BI 設計的地基,先講清楚。

每台設備門檻不同(精密主軸 vs 大型風機差很多),所以 **「3.0 mm/s」對精密主軸是危險、對大型風機卻沒事**。因此:

- **跨機比較/排行一律用正規化指標**:`嚴重度(severity) = 當前值 ÷ 該台危險門檻`(即「離危險還有多少 %」)。
- **狀態(良好/異常/危險)本身就是衍生的可比維度**——它是「值 vs 該台門檻」算出來的(已在 [03 §4](03_資料來源_JSON與SQL設計.md) 定為衍生值,不存死)。

> 一句話:**用「相對門檻的位置」當共同尺規,不要用原始數值。** 這就是把看板從「一堆讀數」變成「風險視圖」的關鍵。

## 3. 假資料設計(擬真,程式化產生)

對應需求 5「程式化產生 10~20 份**擬真**測試檔」。重點是給每台一個**行為樣板**,讓資料會說故事。產生器放 `Common/Helpers/`(測試資料產生)。

### 3.1 行為樣板與分布(以 20 台示意)

| 樣板 | 約占 | 數值行為 | demo 價值 |
|---|---|---|---|
| **健康穩定** Good | ~11 台 | 在低檔小幅波動(zone A/B) | 對照組:正常長怎樣 |
| **緩慢惡化** Warning↗ | ~4 台 | 10 點單調上升,末段越過警告線 | **趨勢圖主秀**:看得到劣化(軸承磨耗/不平衡) |
| **已超標** Danger | ~2 台 | 持續高於危險線 | 風險排行/告警對象 |
| **突發尖峰** Spike | ~2 台 | 平時正常,中間 1~2 點暴衝 | 說明「為何要看趨勢、不能只看當下快照」 |
| **剛上線** zone A | ~1 台 | 極低且平 | 新機基準對照 |

刻意做成 **「多數良好、少數需注意」**——全綠或全紅都不好 demo,這個分布才讓 KPI 與風險排行有意義。

### 3.2 數值要物理合理

- 單位 `mm/s`(RMS 速度);微振場景值落在 ~0.0x–1.x,一般機械可更高。
- 惡化用**單調遞增 + 小雜訊**,不要無故鋸齒亂跳(除非在模擬故障)。
- 時間戳沿用「含時間的量測序列」:做成**每日一筆(實作 40 天,5/23~7/1)**或每小時一筆,讓越線時點落在視窗內、且趨勢區間切換有意義差異。

### 3.3 門檻要逐台不同(依設備類型)

| 類型示意 | warning | danger |
|---|---|---|
| 精密主軸 / 量測平台 | 0.30 | 0.50 |
| 一般馬達 / 泵 | 1.8 | 2.8 |
| 大型風機 / 空壓機 | 2.8 | 4.5 |

逐台不同**正是第 2 節「跨機要正規化」的來源**,也讓「門檻可逐台設定 / 管理者可編輯」這件事有意義。

## 4. 該呈現哪些項目(角色 × 層級)

由粗到細鑽取:總覽 → 明細 → 管理。

### A. 總覽看板(使用者,`/`)

- **KPI 彙總卡**:總台數、良好/異常/危險各幾台、**健康率 %**、當前最高風險設備。
- **狀態分布圖**(甜甜圈/堆疊長條):三色占比一眼看完。
- **設備卡片網格**:名稱、當前值、單位、狀態燈、**距危險門檻 %**、(選配)迷你趨勢 sparkline。
- **風險排行 Top N**:依 `嚴重度 = 值/危險門檻` 降序——直接點名要關注的設備(actionable)。
- **即時指示**:SignalR 連線狀態 / 最後更新時間。

### B. 設備明細(使用者,點卡片進入)

- **ECharts 趨勢圖**:值 over time + 警告/危險門檻線 + 分區色帶 + 可移動游標 tooltip(對應需求 6)。
- **設備 metadata**:名稱、位置、類型、單位、門檻、圖檔(走獨立 API + ETag,見 [03 §5](03_資料來源_JSON與SQL設計.md))。
- **統計摘要**(後端算):平均、最大、最小、標準差、**趨勢方向(斜率)**、視窗內超標次數。
- **近期對比**:最近 N 點均值 vs 前期 N 點均值 → 變化率(惡化/改善)。

### C. 管理(管理者,`/admin`)

- **設備清單表格**:可排序/篩選,欄位含門檻、當前值、狀態。
- **門檻編輯表單**:改 warning/danger(驗證 `warning < danger`)。
- **(Phase 2)** 設備增刪、圖檔上傳、告警歷史。

## 5. 篩選 / 排序 / 分析

### 5.1 篩選與排序維度

- **篩選**:狀態(良好/異常/危險)、區域/產線、設備類型、名稱關鍵字。
- **排序**:當前值、嚴重度(%門檻)、**趨勢斜率(惡化最快優先)**、名稱。

### 5.2 分析指標(公式以後端為準;明細讀數頭依選取區間前端重算)

| 指標 | 算法 | 用途 |
|---|---|---|
| 狀態 | 最新值 vs 該台門檻(`≥` 含;`≥危險→Danger、≥警告→Abnormal`) | 狀態燈 / 篩選 |
| 嚴重度 | 當前值 ÷ 危險門檻 | **跨機排行** |
| 趨勢方向 | 線性回歸斜率(實作採此;非「後半 − 前半」) | 找「正在惡化」的機台 |
| 超標次數 | 視窗內 `value >= 警告門檻`(**含 ≥**)的點數 | 穩定度 |
| 平均 / 最大 / 標準差 | 視窗內算術平均 / 最大值 / **母體**標準差 √(Σ(值−平均)²÷N) | 穩定度 / 摘要 |
| 健康率 | round(良好台數 ÷ 總台數 × 100) | 總覽 KPI |
| 變化率 | (近期均值 − 前期均值) ÷ 前期均值 | 惡化 / 改善 |

> **「視窗」= 明細圖選取的時間區間(全部/7天/30天/自訂)**。後端 `MeasurementAnalyzer` 對整段序列算一份(API 隨明細回傳、為公式正本);明細 Modal 的讀數頭則在**前端依目前選取區間重算**(`trendChart.js` 的 `computeStats`/`slopePerDay`,**鏡像**後端公式,如同 `Shared.evalStatus` 鏡像 `StatusEvaluator`)——讓「平均/最大/標準差/趨勢/超標」**跟著區間變動**;切「7天」就只統計這 7 天。**狀態/最新值**例外:固定取最新一筆(設備當前狀態,與區間無關)。各值滑鼠移上去有 `data-tooltip` 說明公式(對齊本表)。

#### 5.2.1 趨勢方向:算法與三態判定(實作細節)

把「快照」升級成「方向」的關鍵指標(§6 趨勢)。**後端算斜率、前端轉金句**(對齊 §8「後端算、前端畫」)。

**後端:每日斜率(最小平方法線性回歸)** — 落在 [`MeasurementAnalyzer.ComputeSlopePerDay`](../src/VibrationDashboard/Common/Helpers/MeasurementAnalyzer.cs)。以「距首筆的天數」為 x、量測值為 y,求最佳擬合直線的斜率:

```
x_i = (t_i − t_0) 換算成天數        y_i = 第 i 筆量測值
slope = Σ(x_i − x̄)(y_i − ȳ) / Σ(x_i − x̄)²        (單位:每日變化量)
```

- **正 = 惡化(往上)、負 = 改善(往下)**;以**整段視窗**擬合,不是「末點 − 前點」。
- 邊界:單點(n < 2)或所有時間相同(零跨度,分母 ≤ 0)→ 回 `0`;斜率統一 `Round` 4 位小數。
- x 用「距首筆天數」而非原始 ticks,避免大數造成浮點誤差。

**前端:斜率 → 金句(三態)** — 落在 [`trendChart.js renderTrend`](../src/VibrationDashboard/wwwroot/js/trendChart.js)。`0.005` 是死區門檻,避免一點雜訊就被判成有方向:

| 條件(每日斜率 slope) | 顯示 | 顏色 |
|---|---|---|
| `\|slope\| < 0.005` | → 平穩 | 綠(good) |
| `slope > 0` | ↗ 每日 +X,惡化中 | 紅(danger) |
| `slope < 0` | ↘ 每日 X,改善中 | 綠(good) |

**「狀態」≠「趨勢」(常見誤會)** — 兩者看的東西不同,所以可以同時「狀態良好、趨勢惡化」:

| 指標 | 看什麼 | 來源 | 性質 |
|---|---|---|---|
| 狀態 | **最新一筆值** vs 門檻 | [`StatusEvaluator`](../src/VibrationDashboard/Common/Helpers/StatusEvaluator.cs) | 瞬時快照 |
| 趨勢 | **整段視窗**的回歸斜率 | `MeasurementAnalyzer` | 長期方向 |

> 例(BEAR-H01):6/14→6/22 從 0.328 連漲到 0.724,僅 6/23 驟降到 0.2。最新值 0.2 < 警告 0.45 → **狀態良好**;但 10 點回歸斜率仍 ≈ +0.018/天 > 0.005 → **趨勢惡化中**。一天的下殺扳不倒九天的上升——回歸是均攤整段,單一離群點影響有限。這正是同時顯示兩者的用意。

### 5.3 查詢 / 分頁 / 每頁筆數(規模考量,分階段)

題目規模是 10~20 台,但**設備數未來可能大量成長**。所以「查詢框、狀態篩選、每頁筆數、資料分頁」這組**清單檢視**功能,設計前要先看清和即時看板的衝突:

> 關鍵:**清單檢視(分頁)、全廠彙總(KPI/排行)、即時 delta** 三者**擴展需求不同**。分頁會讓前端不再持有全量資料,但 KPI/健康率/風險排行**必須對全廠算** → 彙總要有**獨立來源**,不能綁在「當前頁」。**分頁不能弄壞彙總**,是這題的核心取捨。

決策:**分階段 + 預留接縫**(approach C)。

| 階段 | 清單查詢/分頁 | 彙總(KPI/排行) | 即時 |
|---|---|---|---|
| **現在**(本專案 10~20 台) | 前端對已載入全量做 `filter/sort/slice` | 前端 rollup(全量,正確又即時) | SignalR 單台 delta + 重排(照舊) |
| **未來**(約數百台跨門檻) | `GET /api/machines?search=&status=&sort=&page=&pageSize=`,DB 做 WHERE/ORDER BY/OFFSET-FETCH | 改吃 `GET /api/machines/stats` 聚合端點 | 改推 stats 快照 + 當前頁失效訊號(見 [04 §8](04_即時更新_輪詢與SignalR.md)) |

> **先分清楚場景(重要)**:儀表板(`/`)本來就**不是資料表格**——大量設備下它靠**彙總(KPI/分布/健康率)+ 風險排行 Top-N + 篩選到 actionable 子集**(例:只看異常/危險)呈現「**該關注的那幾台**」,不會把上千台攤平分頁。真正要「瀏覽全部 + 搜尋 + 分頁」的是**管理清單頁(`/admin`,§4 C 已規劃的可排序/篩選表格)**。
> → 因此:**dashboard 的查詢/篩選/每頁筆數/分頁,是對「已聚焦卡片網格」的便利操作,前端版完全夠**;後端分頁的真正需求**主要落在清單頁**,不是 dashboard——上表「未來」欄因此主要服務清單頁(dashboard 仍以彙總 + Top-N 扛規模)。

接縫怎麼留(讓未來是「抽換實作」而非重寫):
- 清單查詢收斂成**單一函式** `listQuery({search,status,sort,page,pageSize}) → {items,total}`——今天內部對 AppState 全量做,明天改打 API,**UI 不動**。
- 彙總來源**抽象化**(今=前端 rollup、明天=`/stats`),KPI/排行模組不認得資料怎麼來。
- API 契約**先定**(見 [02 §5](02_系統架構設計.md) 規劃列),規模未到不實作。

篩選維度沿用 §5.1:**狀態 + 名稱關鍵字**現在就能做;**類型/區域**待 §7 加值欄位齊備再開。

> 報告金句:「我為當前 10~20 台出貨(前端篩選分頁、零後端負擔),但把『清單查詢』和『彙總來源』設計成接縫——擴到後端分頁是**抽換實作、不是重寫**;而且我特別點出**分頁不能弄壞 KPI/排行**:彙總要有獨立於『當前頁』的來源。」

dashboard 卡片網格已實作查詢/篩選/每頁筆數/分頁(前端版)。落點:[Index.cshtml](../src/VibrationDashboard/Views/Home/Index.cshtml)(工具列 `#filterbar` + 分頁列 `#pager`)、[dashboard.css](../src/VibrationDashboard/wwwroot/css/dashboard.css)(`.filterbar`/`.pager`)、[dashboard.js](../src/VibrationDashboard/wwwroot/js/dashboard.js)。

作法與考量(對齊 §5.3 的接縫設計):

| 作法 | 考量(為什麼) |
|---|---|
| **兩層分離**:`AppState.machines` = 全量真相;`view{search,status,pageSize,page}` + `listQuery` = 顯示層 | 分頁讓網格只顯示一頁,但全量仍在前端 → KPI/排行/告警照舊正確 |
| **`listQuery` 是唯一清單查詢入口**,回傳 `{items,total,totalPages,page}` | 同形狀即接縫:未來改打 `/api/machines?search=&page=…` 不動 `renderView`/UI |
| **KPI / 風險排行 / 告警仍對 `AppState` 全量算** | 落實「**分頁不能弄壞彙總**」——彙總來源與「當前頁」解耦 |
| **單一重繪入口 `renderView()`**:載入/刷新/即時推播/工具列操作全走它 | 一處 diff,避免分頁狀態與 SignalR patch 互相打架 |
| **告警的 `prevStatus` 改取自 `AppState` 全量**(非 DOM) | 分頁後該台可能不在當前頁、無 DOM 節點;但升級/恢復是**全廠事件**,不能漏記(事件認定規則見 [05 §7.1](05_視覺與互動設計.md)) |
| **排序與 `reorder()`/後端一致**(危險優先、同級 id 升序) | 先全域排序再切頁 → 分頁邊界穩定,不會跳動 |
| 搜尋**去抖 200ms**、單頁不顯示分頁列、篩選無結果給「清除篩選」 | 效能 + 減少雜訊 + 死路有出口(UX) |

> 一句話:**全量留在前端當彙總與即時的真相,`listQuery` 只切出「要顯示的一頁」**;這層接縫讓「現在前端做、未來換後端」是同一個入口的抽換。

## 6. BI 角度(誠實框定規模)

這專案是 **10~20 台、每台 40 筆(每日一筆 × 40 天)**,所以這裡的「BI」是**輕量彙總 + 指標 + 鑽取**,**不是資料倉儲**。該做、能做的:

- **彙總(rollup)**:總覽 KPI、狀態分布、健康率;(選配)階層彙總 廠區 → 產線 → 設備(若加 `area/line` 欄位)。
- **排行(rank)**:用正規化嚴重度排風險 Top N → actionable。
- **鑽取(drill-down)**:總覽 → 設備明細 → 量測點(游標)。
- **切片(slice)**:依狀態/區域/類型篩選看板。
- **趨勢(trend)**:趨勢圖 + 斜率,把「快照」升級成「方向」。

**不該做(過度設計,列未來方向即可)**:多維 OLAP cube、預測性維護模型、百萬級時序——超出題目規模與評分重點。

> 報告講點:「我用『相對門檻的嚴重度』當跨機共同尺規,把看板從『一堆數字』升級成『先看哪台、為什麼』——這是 BI 的 actionable 精神,而不是堆指標。」

### 6.1 健康率走勢(回推算法,已實作前端版)

把「健康率」這個**快照彙總**沿時間軸鋪開成**方向**(落實上面 trend)。落點:[dashboard.js](../src/VibrationDashboard/wwwroot/js/dashboard.js) 的 `Kpi`(`backfillHealth` / `evalStatus`);UI 規格見 [05 §7.2](05_視覺與互動設計.md)。

**指標定義(單一時間點)**

```
健康率% = round( 良好台數 ÷ 總台數 × 100 )
```

「良好」逐台用**各自門檻**判定(前端 `evalStatus` 鏡像後端 `StatusEvaluator`:≥危險→Danger、≥警告→Abnormal、否則 Good)。**不是把各台健康度相加,而是「數良好幾台 ÷ 總數」**。

**走勢的時間點 = 兩段來源**

| 段 | 來源 | 說明 |
|---|---|---|
| 載入時的歷史線 | **回推**每台既有量測序列(`sparkline`) | 每一期把全廠各台的值各自判定 → 數良好 → 算該期健康率 |
| 載入後的延伸 | 每次彙總更新(推播/刷新/增刪)**追加一點** | 反映真實事件後的當下健康率;滑動視窗上限 30 |

**回推的對齊規則(重要)**

- 前端 `sparkline` **只有數值、沒有逐點時間**(摘要 DTO 僅帶最新一筆時間)→ 無法依日期對齊,改**依序列位置從最新端對齊**(各台「倒數第 k 筆」算同一期)。
- 視窗 W = **各台序列長度的最小值**(7~10 取最短),確保**每一期所有台都有值可數**,不會某期缺人;分母用「有序列的台數」。
- 最右點**對齊目前 KPI 健康率**,與快照一致(連續)。
- ⚠️ 因此「一期 ≈ 一天」是**這份假資料每日一筆的巧合**(見 §3.2),程式本身不讀日期;若改每小時取樣,自動變「一小時一期」。

> 報告講點:「健康率走勢是把『良好÷總數』這個 rollup 沿著每台**既有量測**逐期重算,所以**一載入就有歷史方向**、不用等資料累積;那幾台『緩慢惡化』的設備讓線從高往下掉,直接把『全廠在變差』講出來。」

## 7. 加值欄位建議(讓篩選/BI 成立)

題目只要求 name/image/status/measurements;[01 §3](01_需求分析與領域背景.md) 已加值 `unit` + 門檻。為了篩選與階層彙總,建議再加(皆 optional,JSON/SQL 同步,**不動架構**):

| 欄位 | 型別 | 用途 | 必要性 |
|---|---|---|---|
| `type` | string | 設備類型(泵/風機/主軸…),類型篩選 + 門檻分級依據 | **建議** |
| `area` / `line` | string | 廠區 / 產線,階層彙總與篩選 | 建議 |
| `enabled` | bool | 是否納入監看(管理者可停用) | 選配 |

> 最低限度:只加 `type` 就能撐起「類型篩選 + 門檻分級」;`area/line` 是想做階層彙總才需要。

## 8. 後端 vs 前端的分析責任(呼應準則)

- **後端算、前端畫**:狀態判定、嚴重度、斜率、彙總、排行**全部在 Service 算好**,用 DTO 給前端(呼應 [前端 §1 技術妥協](../rules/前端實作準則/01_前端架構與可維護性.md)、[後端分層](../rules/後端實作準則/01_分層架構與專案結構.md))。
- 前端只負責**呈現與互動**;篩選/排序/分頁**現階段**在前端對「已載入的小資料集(10~20 台)」直接做,不必再打 API。**注意**:此前提綁定「全量在前端」——設備量大量成長後要改後端分頁,屆時彙總(KPI/排行)需獨立的 `/stats` 端點(接縫與遷移路徑見 §5.3)。
- **刻意的例外(明細讀數頭統計)**:平均/最大/標準差/趨勢/超標需「**跟著明細圖選取的區間變動**」,而區間是純前端概念、整段量測也已在瀏覽器 → 由前端 `computeStats` **依區間重算**(**鏡像**後端 `MeasurementAnalyzer`,如同 `Shared.evalStatus` 鏡像 `StatusEvaluator`)。後端仍算一份「整段」統計隨 DTO 回傳作公式正本;健康率/嚴重度/排行仍純後端。原則不變:**有跨機/全廠口徑的算後端,純前端視圖概念(選取區間)才在前端鏡像重算**(見 §5.2)。

## 9. 小結與待決

- 假資料用「**行為樣板 + 多數良好少數異常**」的分布,讓趨勢圖與風險排行有故事。
- 跨機比較一律用「**嚴重度 = 值 ÷ 危險門檻**」當共同尺規。
