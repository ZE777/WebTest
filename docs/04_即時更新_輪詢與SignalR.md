# 04 即時更新:輪詢 vs SignalR

> 對應需求 4(檔案更新時自動更新畫面)。這份回答另一個關鍵討論:
> **「複雜資料的 Dashboard 是不是不能用輪詢?」**

## 1. 先修正一個觀念
選 push 還是 poll,**主因不是「資料複不複雜」**,而是:更新頻率、客戶端數量、延遲要求、以及**每次傳多少**。

> 關鍵:複雜資料的問題不在 poll 或 push,而在**不該每次都把整包重傳**。
> 複雜資料就**只傳「變動的那一台」的差異(delta)**——而 delta 推播正是 SignalR 的強項。

## 2. 對照表

| 面向 | jQuery AJAX 輪詢 | SignalR 推播 |
|---|---|---|
| 機制 | 前端每隔 N 秒主動拉 | 後端偵測到變更才推 |
| 沒有變更時 | **照樣發請求**(浪費) | 完全不傳 |
| 延遲 | = 輪詢間隔 | 近即時(<1s) |
| 傳輸量 | 常見是**整批全撈** | **只推變動那一台** |
| 客戶端變多時 | 請求量 = 客戶端 × 頻率,壓力放大 | 持久連線,推一次即可 |
| 環境需求 | 最單純,任何環境都行 | 需 WebSocket(IIS 要開) |

**所以**:正因為這個看板資料多(10~20 台),SignalR 反而**比盲輪詢更省**——盲輪詢是每 3 秒把 20 台全撈一遍且多數時候沒變;SignalR 只在某台檔案變了才推那一台。

## 3. 核心方法:依「資料變動特性」分配傳輸方式
不要對整頁用同一種方式。把資料分類,各用最適手段:

| 資料 | 變動特性 | 傳輸方式 | 理由 |
|---|---|---|---|
| 機器清單(增刪檔) | 罕見、事件型 | SignalR 推 | 只在真的增刪時推 |
| 狀態 + 最新值 | 偶發(檔更新時) | SignalR 推**單台 delta** | 告警要低延遲,不全撈 |
| 機器圖檔(二位元) | 幾乎不變 | HTTP GET 一次 + **ETag 快取** | **完全不輪詢** |
| 完整趨勢序列(明細,40 筆) | 點機器才要 | AJAX 點擊懶載入 | 平時不傳 |

→ **全頁的重資料一律不盲輪詢**:靜態圖檔快取、清單/狀態/數值靠 delta 推、明細點擊才載入。這就是對複雜 Dashboard 該有的解法。

## 4. 推播管線(後端)
```
FileSystemWatcher(監看 App_Data/machines)
   │  Created / Changed / Deleted 事件
   ▼
去抖 Debounce(同一檔 500ms 內多次事件合併;避開寫一半的檔)
   │
   ▼
重讀「那一台」JSON → 重算狀態(對門檻/ISO)
   │
   ▼
MachineHub.Clients.All.SendAsync("MachineUpdated", dto)   // 或 MachineRemoved
   │
   ▼
前端 realtime.js 收到 → 只更新那張卡片(差異更新)
```

**去抖的必要性**:檔案系統常對一次儲存發出多個 `Changed` 事件,且可能在程式寫到一半就觸發。做法:用 `machineId` 當 key 的計時器(debounce 300~500ms),最後一次才處理;讀檔失敗(讀到半截)就略過等下次。

## 5. 降級策略(韌性)

> **先釐清:SignalR ≠ WebSocket。** WebSocket 是一條全雙工連線的**底層協定**;SignalR 是架在它之上的**高階即時函式庫**,額外提供三件 WebSocket 本身不管的事:**傳輸協商**(優先 WebSocket → 不支援退 SSE → 再退 Long Polling,程式碼一行不改)、**自動重連**(`withAutomaticReconnect()`)、**RPC 式呼叫**(`conn.on("MachineUpdated", …)`,不必自己解析訊息框)。所以本案其實有**兩層降級**:

| 層 | 降級鏈 | 由誰負責 |
|---|---|---|
| 傳輸層(SignalR 內建) | WebSocket → SSE → Long Polling | SignalR 自動協商,程式無感 |
| 應用層(本案自寫) | SignalR 整個連不上 → 每 5 秒 HTTP 輪詢 `/api/machines` | `realtime.js` 的 `Poll` |

- 預設走 **SignalR**;若連線失敗(環境不支援 WebSocket / 反向代理擋掉),前端**自動退回每 5 秒輪詢 `/api/machines`**。
- SignalR 用 `withAutomaticReconnect()`,並在 UI 顯示連線狀態(綠點:即時 / 灰點:輪詢中)。
- 兼顧題目寫的「**定時或動態**讀取」——兩條路都實作,預設動態(推播),備援定時(輪詢)。

## 6. 前端更新原則(避免效能反成本)
- **差異更新**:收到單台 delta 只重繪那張卡片,不重建整個看板 DOM。
- **狀態轉壞要醒目**:良好→危險時,卡片做紅色脈衝動畫並把該台排到前面(見 [05](05_視覺與互動設計.md))。
- **告警是前端衍生、不落地**:通道只送「單台當下快照」,前端拿它比 `AppState` 記憶體裡的舊狀態,**狀態轉移才記一筆**告警(認定規則見 [05 §7.1](05_視覺與互動設計.md));後端沒有「告警」概念,DB 也沒有告警表。
- **前端分工(避免誤解)**:即時更新的「**比較判斷**」是 vanilla JS(`prevStatus !== dto.status`),**不是 jQuery 做的**;jQuery 只負責**發 AJAX**(輪詢降級用)與**操作 DOM**(`.text()`/`.append()` 畫卡片與清單,`.text()` 同時防 XSS),狀態保管則歸 `AppState`。
- 明細趨勢圖只在開啟時初始化 ECharts instance,關閉時 `dispose` 釋放。

## 7. 決策框架:該用 push 還是 poll?(取捨記錄)

> 報告時最該講清楚的一段。**選 push/poll 的決定因素,不是「資料複不複雜」**,而是下面這個問題。

### 7.1 先問一句話:**「資料的源頭,能不能主動通知我?」**

| 源頭能否通知 | 該用 | 為什麼 |
|---|---|---|
| **能**(檔案監看、DB 觸發、訊息佇列、你自己掌控的後端事件) | **Push(SignalR)** | 能「反應事件」,延遲低、只傳變動那一台的 delta、沒變就不傳 |
| **不能**(第三方 API、查的是別人的系統、感測器只能被查) | **輪詢(Poll)** | 沒人會 push 給你,只能定期去拉 |

→ **本專案之所以「能」選 push,正是因為源頭(檔案系統)用 `FileSystemWatcher` 偵測得到變更。** 若資料放在一個我們無法監看、只能定期查的外部系統,**即使資料再單純也只能輪詢**——所以決定因素是「源頭能否通知」,不是「資料簡不簡單」。

### 7.2 輪詢不是「WebSocket 的備胎」——它有自己的主場

把輪詢只當成降級是常見誤解。輪詢**正當的主要用途**:
- **拿不到「被通知」的能力**(最大宗):查天氣/匯率 API、第三方服務狀態、CI build 結果(無 webhook)——對方不會主動推,只能輪詢。
- **延遲容忍度高**:像「每 30~60 秒更新統計」就夠,不需要秒級即時。
- **要極簡 / 高相容 / 無狀態**:純 HTTP,任何環境都能跑、好水平擴展、好快取,不必維護持久連線與連線狀態。
- **(才是)WS 不可用時的降級**:這只是它的角色之一,不是存在理由。

### 7.3 中間地帶與「聰明輪詢」
- **SSE / long-polling**:SignalR 連不上 WebSocket 時自動降級走這些。
- **帶 ETag 的輪詢**:輪詢時帶 `If-None-Match`,沒變回 **304**,大幅降低成本(本專案圖檔 API 即如此,見 [03 §5](03_資料來源_JSON與SQL設計.md))。

### 7.4 本專案的最終取捨
- **預設 push(SignalR)**:源頭偵測得到、要低延遲、只推單台 delta,對 10~20 台看板比盲輪詢更省、更即時、更好擴充。
- **降級 poll(每 5 秒)**:僅在 SignalR 連不上(環境不支援 WS / 反向代理擋掉)時啟用,確保韌性,並同時回應題目「**定時或動態**讀取」。

> 一句話記法:**源頭會通知你 + 要即時 → push;源頭不會通知你 / 不需即時 / 要極簡 → poll。**

## 8. 分頁與即時推播的互動(規模化未來工作)

現在(10~20 台):全量在前端,SignalR 把**單台 delta** 推給 `Clients.All`,前端差異更新那張卡 + 重算 KPI/排行——簡單且夠用。

但設備量大量成長 + 清單分頁後,這個模型會破:
- 前端只持有**當前頁**,收到的 delta 多半是**不在當前頁**的機台 → 無從 patch DOM。
- 「危險優先」排序會因數值變動而位移,**單台 patch 無法可靠維持分頁順序**。

屆時的調整方向(配合 [07 §5.3](07_領域資料與BI分析規劃.md) approach C):
- **彙總(KPI/分布/排行)**:後端算好,推 **stats 快照**(或前端定時拉 `/stats`),與「哪一頁」無關 → 永遠對全廠正確。
- **分頁清單**:delta 改成「**當前頁失效訊號**」,前端只在影響當前頁時**重抓該頁**;離頁機台的變動只進彙總、不動 DOM。
- 連線量大時用 **SignalR Groups**(例:依當前篩選/頁分組)只推相關客戶端,而非 `Clients.All`。

> 一句話:**分頁後,即時性從「推每一台」轉成「推全廠彙總 + 當前頁失效」**——彙總永遠全廠正確,清單只在需要時重抓。
